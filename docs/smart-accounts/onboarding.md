# Smart Accounts: Developer Onboarding

This guide explains the concepts behind Stellar smart accounts so you can follow the [Smart Account Kit guide](README.md) and start building.

No prior knowledge of smart accounts, WebAuthn, or Soroban is required. Familiarity with Stellar accounts and transactions is assumed. Code samples use Swift and the public types in this SDK.

---

## 1. The Problem with Traditional Stellar Accounts

A traditional Stellar account is controlled by a single Ed25519 secret key (an `S...` string). Every transaction must be signed with that key. If the key is lost, the account and its funds are permanently inaccessible. Traditional accounts do support basic multi-signature through additional signer keys and weight thresholds, but they lack programmable recovery, spending limits, context-specific authorization rules, and alternative signature schemes such as WebAuthn passkeys. The user (or the wallet application) carries full responsibility for key storage and backup.

---

## 2. What Is a Smart Account?

Soroban is Stellar's smart contract platform. It allows deploying programs (contracts) that run on the Stellar network, hold state, enforce rules, and interact with other contracts and accounts.

A smart account is a Soroban contract deployed on Stellar that acts as an account. Instead of a single secret key controlling all operations, the authorization logic lives on-chain inside the contract. The contract stores authorization rules (called context rules), each containing authorized signers and policies, and decides at execution time whether a given operation is authorized.

The contract can enforce rules that traditional accounts cannot: requiring multiple signatures, limiting how much can be spent per period, or accepting biometric authentication instead of a secret key.

This SDK uses the OpenZeppelin Soroban contract implementation for smart accounts. OpenZeppelin maintains audited, open-source smart contract libraries.

Recovery works through multiple signers. If a passkey is lost, other signers on the account (a backup passkey, a delegated Stellar account, or an Ed25519 key) can still authorize transactions. Passkeys on modern Apple devices also sync across the user's iCloud Keychain, so a new device automatically inherits the credential when the user is signed into iCloud.

Important: if every signer on an account is lost and no recovery signer was configured, the account becomes inaccessible. Adding a backup signer is recommended for production use.

---

## 3. How Passkeys Replace Secret Keys

WebAuthn passkeys are a standard for passwordless authentication built into modern operating systems and browsers. When a user registers a passkey, the device generates a secp256r1 key pair and stores the private key in secure hardware (Secure Enclave on Apple devices, dedicated security processors on other platforms). Passkeys use secp256r1, a different elliptic curve than Stellar's native Ed25519, because the WebAuthn standard requires it. The user authenticates with a biometric (Face ID, Touch ID) or a hardware security key.

A credential ID is a unique identifier generated by the device when a passkey is created. It allows the device to locate the correct private key for subsequent authentication ceremonies.

Smart accounts use contract-based authorization, so signature verification is handled by a verifier contract deployed on-chain. The verifier parses the WebAuthn-specific data (authenticator data, client data hash) and calls Soroban's native secp256r1 host function to verify the signature. When a user authenticates with their passkey, the smart account contract forwards the signature to the verifier. If the signature is valid and all policies are satisfied, the transaction executes.

On Apple platforms, the SDK ships `AppleWebAuthnProvider`, a `WebAuthnProvider` implementation backed by `ASAuthorizationPlatformPublicKeyCredentialProvider` from `AuthenticationServices`. The provider requires iOS 16 or macOS 13 or later, and the host application must declare the `com.apple.developer.associated-domains` entitlement with a `webcredentials:<rpId>` entry plus a matching `apple-app-site-association` file served under the relying-party domain. See [README.md](README.md#webauthn-provider-construction) for the configuration steps.

Users never see or manage cryptographic keys. They authorize transactions with their face or fingerprint, the same way they unlock their phone.

## 4. Key Concepts

The Swift snippets below show how each concept maps to SDK calls. The `kit` value is an instance of `OZSmartAccountKit`, the SDK's main entry point, covered in the [Smart Account Kit guide](README.md).

Before diving in: an **authorization entry** is a signed statement that a specific signer has approved a specific contract operation. Soroban uses these instead of traditional transaction signatures to authorize contract calls. This term recurs in the sections that follow.

You do not need to construct on-chain data structures manually; the SDK handles all encoding. The on-chain representations shown below use Soroban contract value types (`SCVal`): `Vec` is a list, `Symbol` is a short string identifier, `Address` is a Stellar address, and `Bytes` is raw binary data.

### Signers

A signer is an entity authorized to approve transactions on a smart account. The contract stores signers per context rule (explained below). There are three types.

**Delegated**. A Stellar address (G-strkey for accounts, C-strkey for contracts) that authorizes using Soroban's built-in verification. For accounts, this means signing the authorization entry with the account's secret key, which Soroban verifies automatically. For contracts, Soroban invokes the contract's own `__check_auth`. No external verifier contract is needed.

On-chain representation: `Vec([Symbol("Delegated"), Address(address)])`

SDK call:
```swift
let result = try await kit.signerManager.addDelegated(
    contextRuleId: 0,
    address: "GA7QYNF7..."
)
```

**Passkey (External)**. A WebAuthn public key verified by a verifier contract. The signer stores the verifier's contract address alongside a `keyData` blob containing the 65-byte public key from the passkey registration concatenated with the credential ID.

On-chain representation: `Vec([Symbol("External"), Address(verifier), Bytes(publicKey || credentialId)])`

SDK calls:
```swift
// Convenience: registers a fresh passkey and adds it as a signer in one step.
let added = try await kit.signerManager.addNewPasskeySigner(
    contextRuleId: 0,
    userName: "Alice"
)

// Low-level: adds a pre-registered passkey with raw cryptographic material.
let lowLevel = try await kit.signerManager.addPasskey(
    contextRuleId: 0,
    publicKey: secp256r1PublicKey,      // 65-byte uncompressed SEC1 key
    credentialId: webAuthnCredentialId  // raw credential ID bytes
)
```

**Ed25519 (External)**. A 32-byte Ed25519 public key, also verified by a verifier contract. Uses the same on-chain `External` shape as passkeys, but `keyData` contains only the 32-byte public key (no credential ID). Traditional Stellar accounts verify Ed25519 natively, but smart accounts are contracts and use Soroban's contract-based authorization, which requires a verifier contract for any signature scheme, including Ed25519.

On-chain representation: `Vec([Symbol("External"), Address(verifier), Bytes(publicKey)])`

#### Add

SDK call to add an Ed25519 signer to a context rule:
```swift
let result = try await kit.signerManager.addEd25519(
    contextRuleId: 0,
    verifierAddress: "<C-strkey of Ed25519 verifier>",
    publicKey: ed25519PublicKey  // 32 bytes
)
```

See the [Signing](#signing) subsection below for how to authorize transactions once the signer is on-chain.

#### Signing

When a multi-signer ceremony involves an Ed25519 signer, the smart-account contract calls the registered verifier contract during `__check_auth` to validate the signature. The auth digest — `SHA-256(signaturePayload || contextRuleIds.toXDR())` — is computed by the SDK and must be signed with the Ed25519 private key that corresponds to the on-chain `(verifierAddress, publicKey)` entry.

`kit.externalSigners` is the single front door for all Ed25519 signing. Two custody models are available:

**Model 1 — in-memory keypair (key held in process):** register the key at runtime via `kit.externalSigners`.

```swift
let ed25519VerifierAddress = "CAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAD2KM"
// rawSecretKeyBytes must be exactly 32 bytes (the raw Ed25519 seed, not an S-strkey).
// Stores the keypair in memory under the (verifierAddress, publicKey) tuple.
// Returns the 32-byte public key.
let ed25519PublicKey = try kit.externalSigners.addEd25519FromRawKey(
    secretKeyBytes: rawSecretKeyBytes,
    verifierAddress: ed25519VerifierAddress
)
```

**Model 2 — adapter (key managed externally, e.g. a hardware wallet):** supply a conforming `OZExternalEd25519SignerAdapter` via `config.externalEd25519Adapter` at kit construction. The adapter is injected into `kit.externalSigners` and takes precedence over any in-memory keypair registered for the same `(verifierAddress, publicKey)` pair.

```swift
let config = try OZSmartAccountConfig(
    rpcUrl: "https://soroban-testnet.stellar.org",
    networkPassphrase: Network.testnet.passphrase,
    accountWasmHash: "<64-char hex WASM hash>",
    webauthnVerifierAddress: "<C-strkey of WebAuthn verifier>",
    externalEd25519Adapter: myHardwareWalletAdapter
)
let kit = OZSmartAccountKit.create(config: config)
// kit.externalSigners automatically delegates Ed25519 signing to the adapter.
```

See the [API Reference](api-reference.md#external-signer-management) for the full adapter protocol.

Wallet (G-address) signers follow the same two-model symmetry: an in-memory keypair registered at runtime via `kit.externalSigners.addFromSecret(secretKey:)`, or an adapter supplied via `config.externalWallet` at kit construction.

**Pass the signer to a multi-signer method:**

```swift
// Include the ed25519 case alongside passkey and wallet entries in selectedSigners.
let result = try await kit.multiSignerManager.multiSignerTransfer(
    tokenContract: "<C-strkey of token contract>",
    recipient: "GA7QYNF7SOWQ3GLR2BGMZEHXAVIRZA4KVWLTJJFC7MGXUA74P7UJVSGZ",
    amount: "10",
    selectedSigners: [
        .passkey(credentialId: savedCredId, keyData: savedKeyData),
        .ed25519(verifierAddress: ed25519VerifierAddress, publicKey: ed25519PublicKey)
    ]
)
```

The pipeline validates that a signing source is registered for each `ed25519` entry before any RPC call. If no source is registered for a given `(verifierAddress, publicKey)` tuple, the pipeline throws `ValidationException.InvalidInput` with a message that names the missing verifier. Ed25519 signing uses the same auth-digest computation, signer-index resolution, and authorization-payload assembly as the passkey path — both flow through the same multi-signer pipeline. See [How Passkeys Replace Secret Keys](#3-how-passkeys-replace-secret-keys) for the `__check_auth` mechanics that apply to both.

For the full API surface including `addEd25519FromRawKey`, `canSignEd25519For`, `signEd25519AuthDigest`, `removeEd25519`, and the `OZExternalEd25519SignerAdapter` protocol, see the [API Reference](api-reference.md#external-signer-management).

### Context Rules

A context rule is a named authorization configuration that specifies who can authorize transactions and under what conditions. They are called "context" rules because they apply based on the context of the operation, namely the kind of action being performed and the contract being targeted. Each smart account starts with a default rule (ID 0), created during deployment with the initial passkey signer.

Additional rules can be added that activate only for specific scenarios. For example, a rule that fires only when calling a particular token contract, with its own set of signers and policies, independent from the default rule.

Each context rule stores:

- An ID (`UInt32`).
- A type: `.defaultRule`, `.callContract(contractAddress: String)`, or `.createContract(wasmHash: Data)`. `wasmHash` is the 32-byte SHA-256 hash of a compiled contract binary; `.createContract` matches deployment operations targeting that specific binary.
- A name (human-readable string).
- A list of signers (up to 15).
- A list of policies (up to 5).
- An optional `validUntil` ledger number. Rules with an expiration are automatically skipped after that ledger is reached, useful for temporary authorization grants.

When a transaction requires authorization, the contract collects all non-expired rules that match the operation's context type, plus any default rules, and evaluates them in order until one satisfies the requirements. Rules targeting a specific context type are evaluated before default rules, so the default rule acts as a fallback when no specific rule is satisfied.

SDK calls:
```swift
// Add a rule for a specific contract. Signer objects are created with
// OZDelegatedSigner(address:) and OZExternalSigner.webAuthn(...) or
// OZExternalSigner.ed25519(...). See the SDK guide for construction details.
let addRule = try await kit.contextRuleManagerConcrete.addContextRule(
    contextType: .callContract(contractAddress: "<C-strkey of token>"),
    name: "TokenTransfers",
    signers: [delegatedSigner, passkeySigner]
)

// Read on-chain rules. listContextRules returns parsed ParsedContextRule values;
// getContextRule and getAllContextRules return raw SCValXDR.
let rules = try await kit.contextRuleManagerConcrete.listContextRules()
let count = try await kit.contextRuleManagerConcrete.getContextRulesCount()
let raw = try await kit.contextRuleManagerConcrete.getContextRule(id: 0)
```

### Policies

A policy is a constraint enforced when a context rule is used for authorization. Policies are separate Soroban contracts deployed alongside the smart account. The three OpenZeppelin policy contracts (Simple Threshold, Spending Limit, Weighted Threshold) are standard contracts typically deployed once per network and shared across all smart accounts. You do not deploy a new policy contract for each account; you only need the contract address of each policy type you want to use, the same way you need the WebAuthn verifier address. Each context rule supports up to five policies.

**Simple Threshold**. Requires M valid signatures out of N signers. For example, 2-of-3: any two of three signers must approve.

On-chain state: `threshold: u32`.

```swift
let result = try await kit.policyManager.addSimpleThreshold(
    contextRuleId: 0,
    policyAddress: "<C-strkey of simple-threshold policy>",
    threshold: 2
)
```

**Spending Limit**. Limits the total token amount spent within a rolling window of ledgers. For example, 1,000 tokens per day (about 17,280 ledgers at ~5 s per ledger). The spending limit policy intercepts any contract call whose function name is `transfer` and reads the amount from the third argument. In practice that is almost always a token contract's `transfer` function. If a non-token contract happens to expose a function named `transfer`, the policy applies the limit to it as well; scope spending-limit policies to specific token contracts via context rules when this matters. Other function names are not subject to spending limits.

On-chain state: `spending_limit: i128, period_ledgers: u32`.

```swift
let result = try await kit.policyManager.addSpendingLimit(
    contextRuleId: 0,
    policyAddress: "<C-strkey of spending-limit policy>",
    spendingLimit: "1000",
    periodLedgers: 17_280
)
```

**Weighted Threshold**. Each signer has a weight (vote power). The sum of weights from valid signatures must meet or exceed a threshold. For example, signer A weighs 50, signer B weighs 30, signer C weighs 20, and the threshold is 80. A+B and A+C pass; B+C does not.

On-chain state: `signer_weights: Map<Signer, u32>, threshold: u32`.

```swift
let result = try await kit.policyManager.addWeightedThreshold(
    contextRuleId: 0,
    policyAddress: "<C-strkey of weighted-threshold policy>",
    signerWeights: [
        SignerWeightEntry(signer: signerA, weight: 50),
        SignerWeightEntry(signer: signerB, weight: 30),
        SignerWeightEntry(signer: signerC, weight: 20)
    ],
    threshold: 80
)
```

The policy interface is generic: any Soroban contract that implements the required `can_enforce()` and `enforce()` functions can be used as a policy. Install one with `addPolicy(...)`, passing the install parameters as an `SCValXDR`:

```swift
let installParams = SCValXDR.map([
    SCMapEntryXDR(
        key: SCValXDR.symbol("my_param"),
        val: SCValXDR.u32(42)
    )
])

let result = try await kit.policyManager.addPolicy(
    contextRuleId: 0,
    policyAddress: "<C-strkey of custom policy>",
    installParams: installParams
)
```

The install parameters are policy-specific. The custom policy contract defines what parameters it expects during installation.

A typical setup involves a small number of deployed contracts: one smart account per user, one WebAuthn verifier shared across all accounts on the network, and one to three policy contracts (also shared).

### Deployer

The deployer is the Stellar account that creates the smart account contract on-chain. It has two roles during deployment:

1. **Address derivation.** The credential ID is hashed (SHA-256) to produce a salt, and the contract address is derived from the deployer's public key, the salt, and the network passphrase. The same passkey and deployer always produce the same contract address.
2. **Transaction signing.** The deployer signs the deployment transaction as its source account.

After deployment, the deployer has no privileges over the contract. It cannot move funds, change signers, or modify policies. Only the configured signers can authorize operations.

The SDK ships a default deployer derived from `SHA-256("openzeppelin-smart-account-kit")`, where the hash is used as the Ed25519 seed of a deterministic keypair. Because the deployer has no privileges after deployment, the publicly derivable key is not a security concern; it only matters for address derivation and signing the deployment transaction. This default is suitable for testing and simple deployments, and it is interoperable across compatible Smart Account Kit implementations: the same seed string produces the same deployer everywhere. Production wallet applications typically supply a custom deployer (via `deployerKeypair` in `OZSmartAccountConfig`) for attribution and traceability, since the deployer's public key appears on-chain.

When a relayer is configured, the SDK still uses the deployer to derive the contract address and build the deployment transaction, but submits it to the relayer, which wraps it in a fee bump transaction and pays the fees. The deployer account does not pay fees in this case, but it must still exist on the network with the minimum XLM reserve. On testnet the default deployer is automatically funded via Friendbot when needed.

Without a relayer, the deployer pays the deployment fees directly, so you must provide your own funded deployer via `deployerKeypair`.

When a custom deployer is used, an indexer is recommended for wallet discovery, since clients that do not know the deployer keypair cannot derive the contract address independently.

On testnet, accounts are funded via Friendbot at no cost. On mainnet, deployment and contract operations require XLM for transaction fees and contract storage rent.

### Relayer

A relayer is an optional server that submits transactions on behalf of users and pays the network fees. It wraps the user's transaction in a fee bump transaction, sponsoring the fees.

Without a relayer:

- Someone must fund the deployer account to pay deployment fees.
- The smart account must hold XLM to cover transaction fees for every operation.

With a relayer:

- Users never pay XLM fees.
- The relayer pays for deployment and every subsequent transaction.
- Users only need tokens in their smart account for the transfers they want to make.

Relayer authentication and rate limiting are implementation-specific. The SDK submits transactions to the relayer URL and identifies itself through the `X-Client-Name: ios-stellar-sdk` and `X-Client-Version` HTTP headers; the relayer operator is responsible for access control.

SDK configuration:
```swift
let config = try OZSmartAccountConfig(
    rpcUrl: "...",
    networkPassphrase: Network.testnet.passphrase,
    accountWasmHash: "...",
    webauthnVerifierAddress: "...",
    relayerUrl: "https://relayer.example.com"
)
```

### Verifier Contract

[Section 3](#3-how-passkeys-replace-secret-keys) covers how the smart account delegates on-chain signature verification to a verifier contract through `__check_auth`. This section adds the configuration details.

The WebAuthn verifier and the Ed25519 verifier are typically separate contracts, each specialised for its signature scheme, and each external signer stores a reference to its own verifier. The `webauthnVerifierAddress` configuration parameter sets the default verifier used by every passkey signer the kit registers. For Ed25519 signers, the verifier address is supplied per call to `addEd25519(verifierAddress:publicKey:)`.

The WebAuthn verifier contract address is a required configuration parameter:
```swift
let config = try OZSmartAccountConfig(
    // ...other fields...
    webauthnVerifierAddress: "<C-strkey of WebAuthn verifier>"
)
```

### Indexer

An indexer is an optional service that maps credential IDs to contract addresses. It solves a simple lookup problem: given a passkey credential, which smart account contract does it belong to?

Without an indexer, the SDK derives the contract address from the credential ID and the deployer's public key. This works as long as the deployer keypair is known on the client (including when the default deployer is used). But if a custom deployer was used and the client does not have the deployer keypair, it cannot derive the address.

With an indexer, `connectWallet()` falls back to it when derivation under the configured deployer does not find an on-chain contract — typically because the passkey was added as a signer to an existing wallet rather than deploying its own. If the indexer reports the passkey on more than one contract, `connectWallet()` returns `ConnectWalletResult.ambiguous(credentialId:candidates:)` so the application can present a picker.

The kit falls back to a built-in per-network default indexer URL when `indexerUrl` is `nil`:

- Testnet: `https://smart-account-indexer.sdf-ecosystem.workers.dev`
- Mainnet: `https://smart-account-indexer-mainnet.sdf-ecosystem.workers.dev`

Set `indexerUrl` explicitly to override.

SDK configuration:
```swift
let config = try OZSmartAccountConfig(
    // ...other fields...
    indexerUrl: "https://indexer.example.com"
)
```

## 5. End-to-End Lifecycle

Recall that authorization entries (defined in section 4) are how Soroban authorizes contract operations. The smart account contract checks these entries during transaction execution.

### Creating a wallet

1. The user triggers wallet creation in the app (for example, by tapping a Create Wallet button).
2. The operating system shows the WebAuthn registration prompt (Face ID, Touch ID, or a security key insertion).
3. A passkey is created. The device generates a secp256r1 key pair and stores the private key in the Secure Enclave (or equivalent secure hardware). The public key and a credential ID are returned to the app.
4. The public key and credential ID are passed from the platform's WebAuthn API into the SDK through the `WebAuthnProvider`.
5. The SDK computes the deterministic contract address: it hashes the credential ID with SHA-256 to produce a salt, then derives the address from the deployer's public key, the salt, and the network passphrase using `SmartAccountUtils.deriveContractAddress(...)`.
6. The SDK builds a deploy transaction that creates the smart account contract on-chain. The contract is initialised with a default context rule (ID 0) containing the passkey as its sole signer.
7. The deployer signs the deployment transaction. It is submitted either directly to Soroban RPC or through a relayer for fee sponsoring.
8. The smart account contract is now live on Stellar at the derived address. It can receive tokens and authorize transactions.

```swift
let result = try await kit.walletOperations.createWallet(
    userName: "Alice",
    autoSubmit: true,
    autoFund: true,
    nativeTokenContract: "<C-strkey of native XLM SAC>"
)
```

### Sending tokens

Because smart accounts are contracts, not traditional accounts, they interact with assets through Soroban's contract-to-contract interface rather than Stellar's built-in Payment operation. Assets (including XLM) are accessed through token contracts, the Soroban-compatible wrappers around Stellar assets. Smart accounts call the token contract's `transfer` function to move tokens.

1. The app requests a token transfer (the user enters a recipient and amount).
2. The SDK builds a Soroban transaction that invokes the token contract's `transfer` function with the smart account as source.
3. The SDK simulates the transaction to determine what authorization entries are needed.
4. The operating system shows the WebAuthn authentication prompt. The passkey signs a hash of the authorization entry (not the transaction itself, but the specific on-chain authorization payload).
5. The signed authorization entry is assembled into the transaction.
6. The transaction is submitted, either directly to Soroban RPC or through a relayer for fee sponsoring.
7. On-chain, Soroban calls the smart account contract's `__check_auth` function. The contract verifies the passkey signature via the WebAuthn verifier and evaluates all policies attached to the matching context rule (spending limits, thresholds, etc.).
8. If verification passes and all policies are satisfied, the transfer executes.

```swift
let result = try await kit.transactionOperations.transfer(
    tokenContract: "<C-strkey of token contract>",
    recipient: "GA7QYNF7...",
    amount: "10"
)
```

```
App builds tx --> SDK simulates --> Passkey signs auth entry --> SDK assembles tx --> Submit
                                                                                       |
                                                                           On-chain: __check_auth
                                                                           --> Verifier validates signature
                                                                           --> Policies enforced
                                                                           --> Transfer executes
```

### Reconnecting

1. On app relaunch, the app calls `connectWallet()` with default options. The SDK checks for a saved session in the configured `StorageAdapter`.
2. If a valid (non-expired) session exists, the wallet is silently reconnected. No biometric prompt is shown. The SDK loads the credential ID and contract ID from the session and returns `ConnectWalletResult.connected(...)` with `restoredFromSession: true`. Sessions last seven days by default and can be configured through `sessionExpiryMs` in `OZSmartAccountConfig`.
3. If no session exists or it has expired, `connectWallet()` returns `nil`. The app then shows a Connect button and calls `connectWallet(options: ConnectWalletOptions(prompt: true))` when the user taps it, which triggers a WebAuthn authentication.
4. After authentication, the SDK runs a storage -> derivation -> indexer cascade to resolve the contract address and verifies the contract exists on-chain. If the indexer reports the passkey on more than one contract, the SDK returns `ConnectWalletResult.ambiguous(credentialId:candidates:)` so the application can present a picker; the second `connectWallet` call passes the chosen contract address and re-uses the credential ID without a second WebAuthn ceremony.

```swift
if let connection = try await kit.walletOperations.connectWallet() {
    switch connection {
    case let .connected(credentialId, contractId, restoredFromSession):
        // restoredFromSession == true means no biometric prompt was shown.
        break
    case let .ambiguous(credentialId, candidates):
        // `showContractPicker` is your own picker UI -- implement with your preferred sheet or dialog framework.
        let chosen = await showContractPicker(candidates)
        _ = try await kit.walletOperations.connectWallet(
            options: ConnectWalletOptions(credentialId: credentialId, contractId: chosen)
        )
    }
}
```

## 6. What You Need Before Starting

**Deployment targets**. The bundled `AppleWebAuthnProvider` requires iOS 16+ or macOS 13+, which is the first OS version that ships `ASAuthorizationPlatformPublicKeyCredentialProvider` with passkey support. Older deployment targets can still link the SDK, but they must supply a custom `WebAuthnProvider` implementation.

**Apple platform configuration**. The host application must declare the `com.apple.developer.associated-domains` entitlement with `webcredentials:<rpId>`, and the relying-party domain must serve a matching `.well-known/apple-app-site-association` file. On macOS, set `provider.presentationContextProvider` on the `AppleWebAuthnProvider` instance before any registration or authentication call. On the iOS Simulator, and on unsigned macOS test binaries, `KeychainStorageAdapter` additionally requires the `keychain-access-groups` entitlement to access the Keychain.

**Deployed contracts**. For testnet development, pre-deployed contracts are available, so only their addresses and hashes are needed in the app. The smart account WASM binary is uploaded to the network once and referenced by its 64-character SHA-256 hex hash. Individual smart account contracts are then deployed from this WASM. The WebAuthn verifier is a single contract instance shared by all accounts, so it is referenced by its deployed contract address (a `C...` strkey). Policy contracts (Simple Threshold, Spending Limit, Weighted Threshold) are likewise shared C-strkeys. Check the OpenZeppelin Stellar contracts repository at `https://github.com/OpenZeppelin/stellar-contracts` for current testnet addresses; see the [Uploading your own WASM](README.md#uploading-your-own-wasm) section in the kit guide if you need to deploy your own binary.

Testnet contract hashes can change when contracts are upgraded or testnet is reset, and WASM entries may also expire if their TTL is not extended (they can be restored).

**Optional infrastructure**. A relayer server if you want to sponsor transaction fees for users. An indexer server if you want credential-to-contract lookup beyond the default deployer derivation. Both can be added later.

**Storage adapter**. Use `KeychainStorageAdapter` in production. `UserDefaultsStorageAdapter` writes plaintext property lists to the app container and should be limited to non-sensitive scenarios. `InMemoryStorageAdapter` is for tests and ephemeral demos only and explicitly warns that it is not persistent or secure.

**Funded deployer (without a relayer)**. Without a relayer, the deployer pays the deployment fees. Supply a funded `deployerKeypair` in `OZSmartAccountConfig`. With a relayer, the relayer pays the fees and the default deterministic deployer is sufficient.

Once these prerequisites are in place, continue with the [Smart Account Kit guide](README.md) for the API surface and implementation patterns.
